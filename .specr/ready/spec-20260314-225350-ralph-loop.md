# Spec: Ralph loop for implementing specs

## Problem / Goal

Specr can create and manage specs, but has no way to execute them. The user needs a command that takes a finalized spec and ralph-loops Claude Code against it — fresh context each iteration, one task at a time — until all tasks are checked off. This closes the loop from "spec it" to "ship it" without leaving the Specr workflow.

The ralph loop is Geoffrey Huntley's pattern: a bash `while` loop that repeatedly feeds Claude Code the same prompt, letting state persist through the filesystem and git between iterations. Each iteration gets a clean context window, avoiding context rot.

## Relevant findings about the current project

- `bin/specr` is the sole entry point — a ~170-line bash script.
- Specs live in `.specr/<state>/` folders: `draft/`, `ready/`, `in-progress/`, `completed/`.
- Spec files use Markdown checklists (`- [ ]` / `- [x]`) for task tracking.
- The script already has argument routing via a `case` statement and an `ensure_state_dirs` helper.
- Claude Code is invoked via `exec claude --append-system-prompt`.
- The `prompts/` directory exists but is currently empty and unused.

## Technical implementation

### 1. New command: `specr ralph [spec-file]`

#### Spec selection

- **No argument:** Automatically selects the most recently modified `.md` file in `.specr/ready/`. If `ready/` is empty, prints an error and exits.
- **With argument:** Accepts a spec filename (just the basename, e.g. `spec-20260312-140000-billing-seats.md`). Looks for it in `ready/` first, then `in-progress/` (to allow resuming). Errors if not found in either.
- If multiple specs exist in `ready/` and no argument is given, picks the one with the latest mtime (most recently modified).

#### State transitions (automatic)

- On start: move the selected spec from `ready/` to `in-progress/` (if not already there).
- On completion: move the spec from `in-progress/` to `completed/`.
- These transitions are deterministic and do not require user confirmation (unlike Specr mode's interactive transitions).

#### The loop

```bash
SPEC_FILE=".specr/in-progress/<selected-spec>.md"
MAX_ITER=10
ITER=0

while [[ $ITER -lt $MAX_ITER ]]; do
  ITER=$((ITER + 1))
  echo "Ralph iteration ${ITER}/${MAX_ITER}"

  OUTPUT=$(claude --print --append-system-prompt "${RALPH_PROMPT}" \
    "Implement the next task in the spec: ${SPEC_FILE}")

  # Check completion: output ends with COMPLETED (after trimming whitespace)
  # and the tail of the output is short (last 5 lines, under ~200 chars)
  TAIL=$(echo "$OUTPUT" | tail -5)
  if echo "$TAIL" | grep -q 'COMPLETED$' && [[ ${#TAIL} -lt 200 ]]; then
    echo "Ralph finished after ${ITER} iteration(s)."
    # Move spec to completed
    mv "${SPEC_FILE}" ".specr/completed/$(basename "${SPEC_FILE}")"
    exit 0
  fi
done

echo "Ralph hit max iterations (${MAX_ITER}). Spec remains in-progress."
exit 1
```

Key details:
- Uses `claude --print` (non-interactive mode) so each iteration runs to completion and returns output.
- The `--append-system-prompt` flag carries the ralph prompt (implementation instructions).
- Each iteration is a fresh Claude Code session with a clean context window.
- State persists between iterations through the filesystem (spec file checkboxes, code changes, git history).

#### Completion detection

The completion condition is: the **tail** of Claude's output (last ~5 lines) ends with the word `COMPLETED` and is generally short (under 200 characters). This avoids false positives from `COMPLETED` appearing mid-output in some other context.

When the completion promise is detected:
1. The spec is moved from `in-progress/` to `completed/`.
2. The loop exits with code 0.

When max iterations are hit without completion:
1. The spec stays in `in-progress/`.
2. The loop exits with code 1.
3. The user can re-run `specr ralph` to resume (it will find the spec in `in-progress/`).

### 2. The ralph prompt

The ralph prompt is the system prompt appended to each Claude Code invocation. It must instruct Claude to:

1. **Read the spec file** at the given path.
2. **Find the next unchecked task** (`- [ ]`) in the task list.
3. **Implement only that one task** (including its subtasks, if any).
4. **Check off completed items** in the spec file (`- [ ]` → `- [x]`).
5. **Add a brief note** under the completed task in the spec documenting what was done (e.g., files changed, approach taken). This creates an implementation log for future iterations to reference.
6. **Output `COMPLETED`** as the final word if all tasks in the spec are now checked off. Otherwise, end normally.

The prompt should also instruct Claude to:
- Commit after completing each task (on a feature branch, not main).
- Read and respect any existing `CLAUDE.md` in the project.
- Not modify the spec file beyond checking off tasks and adding implementation notes.
- Not reorder or restructure the task list.

The ralph prompt should be embedded in `bin/specr` alongside the existing Specr system prompt, following the same pattern.

### 5. Branch handling

Before starting the ralph loop, the script should check if we're on `main`/`master`. If so, create and check out a feature branch named after the spec (e.g., `specr/<short-name>`). This prevents ralph from committing directly to main, which aligns with the user's git workflow preferences.

## Constraints and assumptions

- Claude Code (`claude`) must be installed and support `-p`/`--print` and `--append-system-prompt` flags. Verified: both flags exist in the current CLI. `--print` runs non-interactive, prints output, and exits. The trust dialog is skipped in `-p` mode per the CLI docs.
- The spec file's Markdown checklist format (`- [ ]` / `- [x]`) is the source of truth for progress.
- Each ralph iteration is fully independent — no shared state beyond the filesystem.
- Max iterations default of 10 is a safety net, not a target. Most specs should complete in fewer.
- The ralph loop is non-interactive. The user does not participate during execution.
- Git must be available. Each task iteration should produce a commit.

## Acceptance criteria

- `specr ralph` picks the latest spec from `ready/` and starts implementing it.
- `specr ralph <spec-file>` implements a specific spec.
- The spec is automatically moved `ready/ → in-progress/` on start.
- Each iteration runs Claude Code with a fresh context and the ralph prompt.
- Claude implements one top-level task per iteration and checks it off in the spec.
- Implementation notes are added to the spec after each completed task.
- When all tasks are done, Claude outputs `COMPLETED` and the spec moves to `completed/`.
- The loop stops after 10 iterations if not complete, leaving the spec in `in-progress/`.
- Re-running `specr ralph` resumes an in-progress spec.
- A feature branch is created if starting from main/master.
- `specr list` shows specs in their correct state throughout the process.
